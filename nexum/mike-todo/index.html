Penetration Test — Punti server #66

Dal report del penetration test (PDF allegato all'issue) restano 3 punti che richiedono intervento lato Apache/server. I punti applicativi (password policy e moment.js) sono stati risolti lato codice.

Punto 2 — Client-side Desync Attack

Il server potrebbe essere vulnerabile ad attacchi di desincronizzazione HTTP. La mitigazione va fatta a livello di web server / reverse proxy.

Azione richiesta: verificare la configurazione di Apache per il parsing delle richieste HTTP (chunked transfer encoding, content-length handling). Considerare l'abilitazione di HttpProtocolOptions Strict.

Punto 3 — HTTP Strict Transport Security (HSTS) mancante

Il server non invia l'header HSTS, che impedisce il downgrade da HTTPS a HTTP.

Azione richiesta: aggiungere in Apache:

Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"

Richiede mod_headers abilitato.

Punto 8 — Configurazione TLS/cipher

Verificare che la configurazione TLS utilizzi solo cipher suite moderne e disabiliti protocolli obsoleti (TLS 1.0, 1.1).

Azione richiesta: verificare SSLProtocol e SSLCipherSuite nella configurazione Apache.

SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1 SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384 SSLHonorCipherOrder on

Certificato SSL Namirial #82

Il codice Nexum contatta il server Namirial per la firma digitale dei contratti. Attualmente la verifica SSL è disabilitata nel codice perché il certificato del server Namirial non è valido.

Codice attuale in NamirialApi.php riga 62:

'ssl' => array( 'verify_peer' => false, 'verify_peer_name' => false, 'allow_self_signed' => true )

Questa è una "pezza" temporanea discussa con Mike a novembre 2025. Il certificato SSL del server Namirial deve essere valido e verificabile.

Azione richiesta: verificare e correggere il certificato SSL sul server che risponde alle chiamate Namirial. Una volta confermato che il certificato è valido, comunicarcelo e noi rimuoviamo la pezza dal codice.

Importante: non rimuoviamo il verify_peer: false senza conferma scritta che il certificato è stato fixato, altrimenti si rompe la firma digitale dei contratti.

Headers HTTP non sicuri #115

Su nexumapp.estendo.it ci sono due problemi di headers HTTP segnalati nel security scan:

Expires — Data non valida

L'header Expires contiene Fri, 01 Jan 1990 00:00:00 GMT che non è un formato valido secondo RFC-7231. Deve essere nel formato Sun, 06 Nov 1994 08:49:37 GMT.

Set-Cookie — Flag Secure mancante

I cookie di sessione vengono inviati senza il flag Secure, il che significa che potrebbero essere trasmessi su connessioni non cifrate.

Azione richiesta: configurare Apache per forzare i flag di sicurezza sui cookie:

# Richiede mod_headers Header always edit Set-Cookie (.*) "$1; Secure; HttpOnly; SameSite=Lax" # Fix Expires header Header always set Expires "0"

Tool di verifica in tempo reale: securityheaders.com
